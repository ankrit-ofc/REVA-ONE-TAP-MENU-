"TASK: Implement the customer-facing menu category filter as CASCADING CHIP 
ROWS (Option A). Option B (replace-in-place) is documented at the bottom as 
a fallback — read it, but only implement it if Option A proves visually 
heavy in our UI (see "WHEN TO SWITCH TO OPTION B").

CONTEXT:
The menu data is already available as a single nested JSON tree from 
GET /menu (categories with children[] and products[], products with 
variants[]). The filter is pure client-side state against this cached 
tree — no API calls on chip taps. Explore the existing frontend first 
and match its framework, styling system, and component conventions.

============================================================
OPTION A — CASCADING CHIP ROWS (implement this)
============================================================

STATE MODEL:
- Single piece of state: path = array of category ids (or names), e.g. 
  [] (nothing selected = All), ["beverages"], 
  ["beverages", "domestic-liquors", "whiskey"].
- Everything else (which rows render, which chips are selected, which 
  products show) is DERIVED from path + the menu tree. No duplicated state.

CHIP ROWS RENDERING:
- Row 0 (fixed, always visible): "All" chip + all top-level categories. 
  Slightly bolder styling than deeper rows (it's the primary nav).
- For each node along the path that has children, render one additional 
  row below: an "All" chip + that node's children as chips.
- Stop rendering rows when the deepest selected node is a leaf (no 
  children) or nothing is selected at the current deepest row 
  (its "All" is active).
- Each chip shows a product count: "Whiskey (8)". Count = total products 
  in that node's entire subtree (recursive).
- Hide chips for categories whose subtree contains zero active products.
- Selected chip in each row gets filled/active styling; "All" is the 
  default selection for every row.
- Rows scroll horizontally on overflow (no wrapping); hide scrollbars; 
  when a row's selection changes or a new row appears, scroll the 
  selected chip into the horizontal viewport and the new row into the 
  vertical viewport (smooth, respect prefers-reduced-motion).
- Animate new rows in with a brief slide/fade (~150ms); no animation 
  on removal beyond instant collapse.

TAP BEHAVIOR (the core invariant):
- Tapping a chip at row level N: path = path.slice(0, N) then push the 
  tapped category. All rows below level N are discarded/re-derived.
- Tapping "All" at row level N: path = path.slice(0, N). 
- Tapping the already-selected chip toggles back to that row's "All" 
  (i.e., same as tapping "All" at that level).
- Every tap re-filters the product body IMMEDIATELY. Deeper rows are 
  optional refinement, never a required step. The body must never be 
  empty while waiting for a deeper selection.

PRODUCT BODY:
- Shows ALL products in the subtree of the deepest selected node 
  (or the whole menu when path is empty). Flatten recursively.
- GROUPED RENDERING: when the current subtree spans more than one 
  immediate subcategory, group products under sticky section headers 
  named after each immediate subcategory (mirroring a printed menu). 
  When the selection is a leaf or has products only, render a flat list.
- Ordering: categories and products by sort_order then name, matching 
  the tree.
- Each product row: name + variant names with prices (e.g. 
  "60ML 500 · QTR 1300 · HALF 2500 · FULL 4700"). If a product has 
  exactly one variant named "Regular", show just the price.
- Show a small context line above the list: the selected path as 
  breadcrumb text + item count (e.g. "Beverages › Whiskey · 8 items").

EDGE CASES:
- Category with BOTH children and direct products: its direct products 
  appear in the grouped body under a header with the category's own 
  name (or "Other" — match existing convention), listed after the 
  subcategory groups... or before, pick one and be consistent.
- Deactivated categories/products/variants never render (server already 
  filters, but guard client-side too).
- If the tree is only 1 level deep anywhere, this degrades gracefully 
  to a plain single chip row — no special-casing needed.
- Deep trees (4+ levels) still work; rows just stack. See Option B note.

ACCEPTANCE CHECKS (Option A):
1. Tap "Beverages" -> body shows all beverage products grouped by 
   subcategory AND row 2 appears with Domestic liquors / Soft drinks etc.
2. Tap "Domestic liquors" -> body narrows, row 3 appears (Whiskey/Rum/
   Vodka/Gin).
3. Tap "Whiskey" (leaf) -> body shows only whiskey products, NO new row.
4. With path 3 deep, tap "Pizza" in row 0 -> rows 2 and 3 disappear, 
   body shows pizzas.
5. Tap "All" in row 2 -> path truncates to ["beverages"], row 3 gone, 
   body widens back to all beverages.
6. Counts on chips match visible products; zero-product categories hidden.
7. No network requests fire on any chip tap.
8. Works at 360px viewport width with horizontal chip scrolling.

============================================================
OPTION B — REPLACE-IN-PLACE (fallback / sidegrade, do NOT build now)
============================================================

WHEN TO SWITCH TO OPTION B:
Consider switching only if, with real menu data, Option A regularly 
renders 3+ stacked rows and pushes products below the fold on typical 
phones, or the client finds the stacked rows visually heavy. Because 
both options share the same path[] state model and derived-subtree 
filtering, switching is a rendering-layer change only — do not change 
state management or data flow.

BEHAVIOR SUMMARY:
- Row 0 stays fixed (top-level categories), exactly as in Option A.
- Only ONE dynamic row ever exists below it. It always shows the 
  children of the deepest selected node.
- Selecting a chip in the dynamic row that has children REPLACES the 
  dynamic row's chips with that node's children (path grows, row 
  content swaps).
- The selected path renders as a compact breadcrumb between row 0 and 
  the dynamic row: "Beverages › Domestic liquors ✕" — tapping a crumb 
  truncates the path to it; tapping ✕ clears back to the row-0 selection.
- All other behavior (immediate filtering, subtree product body, 
  grouped headers, counts, hiding empties, leaf = no children to show) 
  is IDENTICAL to Option A.
- Tradeoff to note in code comments: saves vertical space; costs one 
  extra tap for sibling navigation at deep levels (must go up via 
  breadcrumb, then sideways)."

  WITH RECURSIVE subtree AS (
    -- Anchor: the starting row
    SELECT id, parent_id, name, 0 AS depth
    FROM categories
    WHERE id = 42                      -- 'Beverages'

    UNION ALL

    -- Recursive step: children of whatever's already in subtree
    SELECT c.id, c.parent_id, c.name, s.depth + 1
    FROM categories c
    JOIN subtree s ON c.parent_id = s.id
)
SELECT * FROM subtree;
