Security Concerns and Mitigations for Payment Integration

1. Secret Leakage
   Issue:
   Payment gateway secrets (eSewa, Khalti, Fonepay) may be exposed by placing them in frontend code, committing `.env` files to version control, logging them, or returning them in API responses. Exposure of these secrets allows attackers to forge signatures, impersonate the backend, and bypass payment verification.

Mitigation:

* Store all gateway secrets only on the backend using environment variables or a secure secret manager.
* Never expose secrets through frontend environment variables (e.g., VITE_*).
* Exclude `.env` files from version control using `.gitignore`.
* Redact secrets from logs and error messages.
* Restrict access to production secrets to authorized personnel only.

2. Missing Signature/Verification Checks
   Issue:
   Trusting payment status values received from the frontend or callback payloads without verification can allow attackers to mark invoices as paid without completing a legitimate transaction.

Mitigation:

* Verify all gateway signatures exactly as defined by official documentation.
* For Khalti, always perform server-side lookup using the secret API key and confirm the payment status from Khalti before updating invoice status.
* Reject any callback or payment update that fails verification.
* Treat the frontend as an untrusted source of information.

3. Replay Attacks
   Issue:
   Attackers may resend previously valid callbacks or reuse old payment responses to trigger duplicate payment processing.

Mitigation:

* Enforce unique constraints on `transaction_uuid`, gateway transaction IDs, and invoice-payment mappings.
* Store processing metadata such as `processed_at`.
* Reject callbacks for payments that have already been finalized.
* Ensure payment state transitions are allowed only once.

4. Race Conditions and Concurrent Processing
   Issue:
   Simultaneous callbacks, duplicate requests, or multiple user actions may attempt to process the same invoice at the same time, leading to inconsistent states or duplicate side effects.

Mitigation:

* Wrap payment processing in database transactions.
* Lock affected records using mechanisms such as `SELECT ... FOR UPDATE`.
* Use unique database constraints to prevent duplicate records.
* Recheck invoice status within the transaction before committing changes.

5. Trusting Frontend Actions
   Issue:
   Allowing frontend events (success page visits, query parameter manipulation, JavaScript modifications, etc.) to directly mark orders as paid, close invoices, reset tables, or trigger business actions enables users to bypass payment requirements.

Mitigation:

* The backend must be the sole authority for payment validation and state transitions.
* Update invoice status, close orders, free tables, issue receipts, and trigger downstream actions only after successful backend verification.
* Use frontend components only to display payment status and initiate requests, never to determine payment outcomes.
* Prefer direct gateway-to-backend verification flows whenever supported.
